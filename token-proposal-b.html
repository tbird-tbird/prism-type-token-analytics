Base Tokens (Proposal B)

Proposed changes to the base type token architecture · Source: Prism Base (Mp8dBTFCjNZsLGrG4A5O9V)

Executive Summary

Compatibility
Naming continuity
Font-size, letter-spacing, and line-height tokens retain their existing index-based names, minimising refactoring work and maintaining continuity with the current system.
Precision
One token per typeface
Each brand and system font gets its own dedicated token, giving teams exact control over which typeface is used and where.
Consistency
Standardised font weights
Font weights now use universal numeric values, ensuring the same weight renders consistently across all platforms and tools.
Hygiene
Retiring unused tokens
10 tokens with zero usage across all 32 product modes have been removed, keeping the system lean and easier to maintain.

base/font-family

base/font-family/brand is renamed to individual per-font tokens — one for each brand typeface. This keeps base tokens as pure, factual named constants and moves brand intent to the usage/semantic layer, where a token like font-family/display can reference whichever base font is appropriate per brand. Individual monospaced tokens are also added for each brand and system font to support code and tabular content.

base/font-weight

Added medium (500) to fill the gap between regular (400) and semibold (600), completing the weight scale. All values are also switched from strings (e.g. Regular, Bold) to numeric CSS font-weight values for cross-platform consistency.

Why numeric values? The same numeric weight (e.g., 800) may be labelled differently across fonts — "Extra Bold" in Inter, "Heavy" in SF Pro. Numeric values are unambiguous and align with the W3C Design Tokens spec, which treats strings as aliases for numeric weights.

base/letter-spacing

The letter-spacing scale has been revised for better distribution. Token naming retains the existing index-based system. Two new tokens are added: step 5 (−1) and step 60 (−0.1). Key value changes: −0.5 → −0.6 at step 20; −0.01 → 0 at step 70.

Scale distribution: The current scale has uneven gaps: −2, −0.8, −0.5, −0.4, −0.3, −0.2, −0.01. The proposed scale fills the gaps more evenly: −2, −1, −0.8, −0.6, −0.4, −0.3, −0.2, −0.1, 0. This provides finer control in the −0.2 to 0 range where most body text letter-spacing lives.

base/line-height

Token naming retains the existing index-based system. The scale is extended at the upper end to support larger display sizes, and base/line-height/55 (38px) is removed. Five new steps are added (80, 95, 100, 140, 150).
