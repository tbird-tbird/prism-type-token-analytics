Prism Type Token Architecture Proposal

Evolving from a 3-layer system to a 5-layer architecture by introducing a dedicated Base Brand primitive for typefaces and a Composite layer for Type Styles between Usage and Component.

Note on the Composite layer: Composite Type tokens (Type Styles) already exist within the Usage layer today, sitting alongside primitive usage tokens. This proposal pulls them out into their own dedicated Composite layer — keeping things tidy and mirroring the same structural pattern being proposed for color tokens.

Known issues & what this solves

Known issues

✕
Semantic aliases at Base are 100% unused. 10 font-size aliases (xx-small → xxxxx-large) exist at the Base level but are never referenced by any token across all 32 modes. They duplicate naming that belongs at the Usage layer.
✕
17 of 57 base tokens (30%) are never referenced. Nearly a third of the base type token inventory is dead weight with no clear path to removal because layer ownership is ambiguous.
✕
Composite type tokens are mixed into Usage. Type Style tokens (usage/type/*) already exist — display, title, body, label, link — but they live alongside primitive usage tokens in the same layer, with no structural separation from scalar values like font-size and line-height.
✕
Font-family has no structural home. Brand typefaces are embedded in Usage alongside semantic scale values, with no distinction between identity-level choices and functional type decisions.
✕
Base is completely private. Even brand-critical typefaces that partners need to reference are inaccessible — forcing teams to hardcode values or misuse the Usage layer.
✕
Two naming conventions coexist in Base. Numeric steps (font-size/25) and semantic aliases (font-size/small) describe the same concept with no clear convention, creating confusion about which to reference.

What this solves

✓
One naming convention per layer. Semantic aliases are removed from Base. Numeric steps stay at Base, semantic names stay at Usage — one convention, one layer, no ambiguity.
✓
17 unused tokens can be deprecated with confidence. Clear layer responsibilities make the audit trail unambiguous. Tokens with no layer-appropriate consumers can be removed safely.
✓
Type Style layer reorganizes existing composites. The usage/type/* tokens already exist — this proposal moves them into their own dedicated layer, cleanly separating composite type styles from primitive usage tokens and giving them a clear structural home.
✓
Base Brand gives typefaces a dedicated home. Brand fonts are accessible to designers and engineers without conflating brand identity with functional semantic values at the Usage layer.
✓
Base Brand is public by convention, not by enforcement. Figma publishes at the collection level — Base Brand can't be hidden within a shared collection. By documented convention, font-family tokens should only be consumed through Usage tokens, never referenced directly. This intent is enforced at the pipeline level (Style Dictionary), not by Figma.
✓
Scale audits become simpler. A numeric-only Base with clean step coverage is easier to understand, document, and evolve — especially across 32 theme modes.

Migration path

Two token groups are moving to new layers. Any Figma components or code referencing these paths will need updating. Recommend maintaining aliased tokens at old paths during a deprecation window before hard removal.

Requires migration
Composite type tokens
The usage/type/* variable group moves out of the Usage layer and into the new Composite layer. Any component or code referencing these tokens by path will break.
usage/type/* → type/*
Requires migration
Font family tokens
The usage/font-family/* variable group moves from the Usage layer into the new Base Brand layer. References in components and code will need to be updated.
usage/font-family/* → base-brand/font-family/*
No migration needed
Base semantic aliases
The 10 semantic aliases at Base (xx-small → xxxxx-large) have zero references across all 32 modes. They can be removed without touching any consumer.
base/font-size/xx-small → removed
No migration needed
17 unreferenced base tokens
30% of the base token inventory has no consumers across any mode. These can be deprecated and removed with no downstream impact.
base/* → deprecated

Recommended approach: When moving usage/type/* and usage/font-family/* to their new layers, keep aliased tokens at the old paths for one release cycle with a deprecation notice. This gives consuming teams time to update references before the old paths are removed.

Reference

What is a Type Style token?

A composite token that bundles font-size, line-height, letter-spacing, and font-weight into a single reusable typography definition.

i.e. type/display/large, type/title/medium, type/body/default, type/label/small/strong

Rules
  1. Type Style tokens are public and can be referenced by designers and engineers
  2. Type Style tokens reference Usage tokens — never Base directly
  3. Component tokens should reference Type Style tokens rather than bundling properties individually
  4. Changes to a Type Style token have a predictable, auditable blast radius
  5. Type Style tokens are not brand-specific — brand differentiation lives in Base Brand
