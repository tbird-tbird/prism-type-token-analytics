Usage Tokens

Proposed changes to the usage type token architecture · Source: Kinship: Typography (y2JmgMCEIS5bZb288qCsYv)

Executive Summary

Abstraction
Semantic naming over raw values
Token names describe intent — usage/font-size/small, usage/font-weight/strong — rather than raw values. Teams can apply tokens without memorising the base scale.
Themeability
Per-theme resolution
Consumer and Courier+ resolve the same usage tokens to different base values, enabling distinct type scales per brand from a single shared token set.
Architecture
Clean two-hop alias chain
Type-style tokens reference usage property tokens, which reference base tokens. No direct base references in composite styles — the chain is type-styles → usage → base.
Simplicity
22 type styles reduced to 11
The current Prism composite token set contains 22 type styles. This proposal halves that to 11 — drastically reducing the options designers need to choose from and the likelihood of misuse across products.

Manual Overrides
Token architecture: The usage layer introduces two tiers. Property tokens (usage/font-size, usage/font-weight, etc.) give semantic names to base scale values and are independently themeable. Type-style tokens (usage/type-styles) compose those property tokens into complete per-style configurations for Consumer and Courier+.

usage/type-styles

11 semantic type styles, each defined as a set of 6 property tokens (font-family, font-weight, font-size, letter-spacing, line-height, text-decoration). All property tokens reference the usage layer — no direct base token references — maintaining a clean type-styles → usage → base alias chain.

Text-decoration values are not yet documented in detail. Current values: none for all styles except link variants, which use underline.

Manual overrides: Three Courier+ type-style tokens carry a raw value of 22 for font-size. This value does not map to any token in the foundational font-size scale and is therefore a direct numeric override. These are marked with the Override tag below.
