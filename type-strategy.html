Type Foundation Consolidation

Typography · Unified Design Foundations

This document outlines the strategy to refactor Prism's base and usage type tokens into a unified, scalable typographic foundation that supports all brands and products across DoorDash, Wolt, Deliveroo, and future products.

Program Context
Phase 2: Unified Design Foundations
Following Phase 1 (Pedregal), which unified brand asset ownership under Prism, Phase 2 focuses on consolidating duplicate primitives across Color, Typography, Icons, Spacing, Elevation, Motion, and Illustration. Type is one of seven property families being addressed. Semantic tokens remain stable throughout — only the primitive and structural layer changes.

Problems

✕
System doesn't support product vision

The current type system no longer provides the expressive range our products need to realize their design vision, constraining product teams rather than enabling them.

✕
Too many styles leads to misuse

An excessive number of type styles makes it too easy to misuse the system, creating visual inconsistencies across products and undermining the coherence of the design language.

✕
Brand fonts introduce accessibility risk

Using brand fonts for body text introduces accessibility and legibility challenges compared to platform-standard system fonts, which are optimized for reading at smaller sizes.

✕
Scale and token structure has redundancies and gaps

Redundancies and gaps in the existing type scale and token structure negatively impact usability, scalability, and long-term maintainability — especially across 32 theme modes.

Goals

Design and implement a simplified, scalable, and accessible typography system that serves all products, brands, and surfaces across the DoorDash portfolio.

✓
Reduce cognitive load and misuse Offer a clear and intentional set of type styles for each surface — fewer options, better outcomes, less room for inconsistency.
✓
Enable product vision with expressive range Give product teams sufficient expressive flexibility to realize their design vision without stepping outside the system.
✓
Improve accessibility and legibility Leverage system fonts for body text to optimize readability across sizes and surfaces, reducing reliance on brand fonts where they create risk.
✓
Establish a rationalized token architecture Create a clear, layered token structure that supports consistency, scalability, and long-term maintainability as the system evolves toward Kin.

How the token architecture delivers on this

Proposal
Evolving from a 3-layer to a 5-layer type token architecture — view proposal →
Reduce misuse Dedicated Composite layer separates type styles from primitive tokens, making the intended consumption pattern structurally explicit. Component tokens reference one Type Style instead of bundling properties ad-hoc.
Expressive range Cleaned-up Base layer removes dead weight (17 unreferenced tokens, 10 semantic aliases) and establishes a clearer numeric scale — making it easier to reason about coverage and add intentional steps where gaps exist.
Accessibility Base Brand layer makes the architectural intent explicit: brand fonts are identity values, not defaults. Separating them from semantic usage tokens creates the structural foundation for directing brand fonts to display contexts and system fonts to body contexts.
Token architecture 5-layer architecture (Base · Base Brand · Usage · Composite · Component) establishes one canonical primitive per property, clear public/private boundaries, and a stable semantic layer — directly meeting Phase 2's program goals and laying the groundwork for Kin's language elevation.
